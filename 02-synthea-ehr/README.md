# Project 2: Synthetic EHR Analysis (Synthea)

## Business Question

Using a realistic, multi-table electronic health record structure, what patterns exist in patient encounters, diagnoses, medications, and procedures — and what does this reveal about care patterns, cost, and utilization across a synthetic patient population?

Unlike Project 1 (a single wide table), this project models a genuine relational EHR: patients, encounters, and three encounter-linked tables (conditions, medications, procedures), connected by real foreign key relationships — closer to how hospital data warehouses are actually structured.

## Dataset

- **Source:** [Synthea Dataset, Kaggle](https://www.kaggle.com/datasets/sameeralamynwa/synthea-dataset) — synthetic patient records generated by [Synthea](https://synthea.mitre.org/), an open-source patient population simulator built by MITRE, informed by real clinical and epidemiological data.
- **Tables used:** `patients`, `encounters`, `conditions`, `medications`, `procedures` (5 of 18 available files — scoped down intentionally to keep the project focused on core clinical/utilization analysis rather than billing/organizational detail)
- **Population:** 1,183 patients, 69,121 encounters, 43,595 conditions, 69,581 medication records, 115,266 procedures
- **Data is fully synthetic** — no real patient information; free of privacy/legal restriction, but also not reflective of any real hospital's actual population or outcomes

**Known limitations (noted upfront):**
- Synthetic data reflects the assumptions of Synthea's disease/care models, not a real-world population — findings describe patterns in this generated dataset, not real clinical epidemiology
- `organization_id`, `provider_id`, and `payer_id` are retained as reference columns but not joined to lookup tables (organizations/providers/payers were out of scope for this project)
- High average encounters-per-patient (~58) reflects Synthea's generation of complete lifetime medical histories per patient, not real-world visit frequency

## Approach

Built a normalized relational schema in PostgreSQL: `patients` and `encounters` as parent tables, with `conditions`, `medications`, and `procedures` each referencing `encounters` (and `patients`) via foreign keys. `conditions`/`medications`/`procedures` use surrogate `SERIAL` primary keys, since the source data has no natural single-column unique identifier for these tables; `patients` and `encounters` use their native UUIDs.

During initial loading, a data source mismatch was identified: an earlier `patients.csv` had been sourced from a different Synthea generation run than the other four files, resulting in zero valid patient references across all 69,121 encounters. This was diagnosed through systematic elimination (ruling out encoding, whitespace, type mismatches, and session state before confirming the population itself didn't overlap), and resolved by re-sourcing `patients.csv` from the same dataset as the other four files. Foreign key constraints — dropped temporarily to allow diagnosis — were re-added once the corrected data loaded cleanly. Documented in `sql/03_data_cleaning.sql`.

A minor UTF-8 encoding issue in the patient name fields (non-standard characters in a small number of rows) was also resolved via `iconv` conversion prior to loading.
